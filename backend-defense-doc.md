# YoEasy 后端答辩文档（数据库小白友好版）

## 1. 我的职责与答辩口径

我是本项目后端开发负责人，主要完成：
1. FastAPI 接口开发
2. 用户认证与权限控制（JWT）
3. 数据模型设计与数据库读写（SQLAlchemy + SQLite）
4. 文件上传与静态资源访问
5. 商品、分类、收藏、消息、购物车等业务接口

答辩开场建议：
“我主要负责后端接口和业务逻辑，数据库部分通过 ORM 实现了可维护的数据读写，保证了完整业务闭环。”

---

## 2. 技术栈与为什么这么选

1. FastAPI：开发快、结构清晰、接口联调效率高。
2. SQLAlchemy ORM：用 Python 类映射数据表，降低手写 SQL 的复杂度。
3. SQLite：项目阶段部署简单，适合快速开发和演示。
4. JWT + bcrypt：登录态无状态管理，密码做哈希更安全。

---

## 3. 后端架构（可直接讲）

后端采用三层思路：
1. 路由层：接收请求、参数校验、返回结果。
2. 业务层：处理权限判断和业务规则。
3. 数据层：通过 ORM 操作数据库。

数据流：前端请求 -> API 路由 -> 业务处理 -> 数据库 -> 返回 JSON。

---

## 4. 数据库部分怎么讲（重点）

## 4.1 给老师的通俗解释
数据库可以理解为“结构化的数据仓库”，把用户、商品、购物车这些核心业务数据长期保存起来。

## 4.2 项目里主要表

### users（用户表）
作用：保存账号和权限。
核心字段：
- id（主键）
- username（唯一）
- hashed_password
- role（user/admin）
- favorites（当前实现为逗号分隔字符串）

### products（商品表）
作用：保存商品信息。
核心字段：
- title、price、category
- seller、sellerContact
- status（active/sold）
- publishDate

### categories（分类表）
作用：管理商品分类。
核心字段：
- name
- value（唯一）
- sort_order

### cart_items（购物车表）
作用：记录“谁加了什么商品、数量多少”。
核心字段：
- user_id
- product_id
- quantity

### messages（消息表）
作用：用户通知与已读状态管理。
核心字段：
- user_id
- title、description、type
- is_read

## 4.3 我的数据库实现策略
1. 先保证业务闭环，再做结构优化。
2. 用 ORM 显式定义字段和约束，减少低级错误。
3. 对关键字段设置唯一/索引，保证数据可靠性。

---

## 5. 核心后端能力说明

## 5.1 认证鉴权
1. 注册：校验重名 -> bcrypt 哈希 -> 入库。
2. 登录：校验账号密码 -> 生成 JWT。
3. 鉴权：通过 Bearer Token 解析当前用户。
4. 权限：管理员接口校验 role，不通过返回 403。

## 5.2 商品与分类
1. 商品支持查询、发布、编辑、删除。
2. 编辑/删除仅允许卖家本人或管理员。
3. 分类仅管理员可维护。
4. 删除分类前检查是否有关联商品。

## 5.3 收藏、消息、购物车
1. 收藏使用 toggle 接口维护状态。
2. 消息支持列表、单条已读、全部已读。
3. 购物车支持新增、修改数量、删除。

## 5.4 文件上传
1. 提供图片上传接口。
2. 文件写入 uploads 目录。
3. 通过静态资源路由访问。

---

## 6. 我对数据库不熟时如何保证质量

1. 先把业务对象理清（用户/商品/购物车/消息）。
2. 用 ORM 模型驱动数据结构。
3. 每条关键业务链路都做联调验证。
4. 用初始化脚本准备测试数据。

可讲：
“我数据库理论还在补，但在工程层面通过 ORM 和全链路验证保证了功能可靠。”

## 7. 已知不足与改进计划（一定要讲）

### 当前不足
1. favorites 用字符串存储，不利于复杂查询。
2. 购物车查询存在 N+1 查询问题。
3. 上传接口校验还可加强（类型、大小、白名单）。
4. SQLite 适合开发演示，不适合高并发生产。

### 下一步优化
1. 收藏改为中间表（user_favorites）。
2. 购物车改 JOIN 查询，减少数据库往返。
3. 上传增加 MIME/大小校验。
4. 迁移 MySQL/PostgreSQL + Alembic 版本管理。

## 8. 常见追问与标准回答

Q1：为什么选 SQLite？
A：项目阶段优先快速落地，SQLite 部署成本最低，后续可迁移 MySQL/PostgreSQL。

Q2：你怎么理解 ORM？
A：ORM 是对象和表的映射，让我用 Python 对象方式做数据库增删改查。

Q3：你数据库不熟，为什么敢负责后端？
A：我用 ORM + 清晰模型 + 接口联调把系统稳定跑通，并且明确了数据库优化路线。

Q4：怎么保证权限安全？
A：后端接口做角色校验，前端限制只是体验层，真正安全边界在后端。
